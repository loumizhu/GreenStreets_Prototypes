# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A set of clickable HTML/CSS/JS prototypes for "Greenstreets" â€” no build system, no package.json, no server. Each prototype is a **single self-contained HTML file** you open directly in a browser (double-click or drag into a browser tab). There is no dev server, bundler, linter, or test suite in this repo â€” "running" a prototype just means opening the file.

Not a git repository â€” there is no version control here. Treat file edits as final; there's no diff/history safety net, so be extra careful with destructive changes and consider keeping backups (the `old/` folder is exactly that: prior iterations kept for reference).

- **Animated focus outline** (fields, selects & buttons): on focus/click the control's outline is *drawn* in the focus color â€” the stroke starts at the centre of the top & bottom edges and spreads symmetrically around the rounded corners to the left & right sides, completing a full rounded border. It's a **conic-gradient border ring** on a lightweight overlay element (`.fs-ring`) that **`js/greenstreets-theme.js`** creates and positions over whatever control is focused (one at a time). The ring is inserted as an **`position:absolute` sibling of the focused control** (not a fixed overlay on `<body>`) so it shares that control's stacking context â€” it hides behind a modal when the underlying field is behind one, shows above content for a field inside the modal, and scrolls with the control (so `.main`/`.pbody` are `position:relative`). ## Architecture of each prototype file

Every prototype is built the same way â€” a single `<html>` file with all CSS in one `<style>` block and all behavior in `<script>` blocks at the bottom. There's no framework, no modules, no build step: everything is vanilla JS/CSS/HTML in one document.

**Screens, not pages.** Each distinct view (login, list, detail, wizard step, etc.) is a `<div class="screen" id="sN">` (or `pN` for a step in a multi-step flow, e.g. add-retailer steps `p1`-`p4`). Only the screen with class `.on` is visible (`display:flex`); all others are `display:none`. Navigation between screens never triggers a page load â€” it's all client-side visibility toggling via a global `go(id)` function:

```js
function go(id){
  document.querySelectorAll('.screen').forEach(s => s.classList.remove('on'));
  el = document.getElementById(id); if(el) el.classList.add('on');
  // also syncs the .pnb prototype-nav button highlight to match
}
```

Real UI elements (sidebar nav links, buttons, breadcrumbs, table rows) call `go('sN')` via inline `onclick` attributes to move between screens â€” this is the actual navigation UX being prototyped.

**Prototype nav bar (`.pnav` / `.pnb` buttons).** A sticky bar at the very top of the page (outside the real app UI) lists every screen as a button for quick jumping while reviewing the prototype â€” this is a reviewer/demo aid, not part of the product UI being designed. When adding a new screen, add both the `.screen` div and a corresponding `.pnb` button so it's reachable.

**Design tokens via CSS variables.** Each file defines its palette/spacing in a `:root{...}` block (e.g. `--gs` for the Greenstreets green accent, `--nv`/`--nv2`/`--nv3` for navy background shades, `--tw`/`--tw2`/`--tw3` for white-text opacity levels, `--rs`/`--rl` for border-radius scale). Reuse these variables rather than hardcoding new colors/radii when editing a screen â€” check the `:root` block at the top of the `<style>` section of the specific file you're editing, since values drift slightly between prototypes.

**No real data or backend.** All content (retailer names, stats, table rows) is static HTML baked into each screen, or occasionally a small inline JS array (e.g. a country picker's option list). Wizard steps and forms don't persist input between screens beyond simple JS-driven UI feedback (e.g. `selectCountry()` filling in a dependent field) â€” there's no state store.

## v6 admin prototype â€” extra architecture on top of the base pattern

v6 follows everything in "Architecture of each prototype file" above (screens, `go()`, `.pnav`/`.pnb`, CSS-variable design tokens), plus these additions built up over several redesign passes:

**Shared sidebar via JS injection, not copy-pasted markup.** The operator sidebar (logo, nav links, notifications, user footer) is identical across ~14 screens, so instead of duplicating that HTML per screen, each screen just declares an empty mount point:
```html
<div class="sidebar" data-active="s11"></div>
```
`sidebarHtml(activeId)` builds the full sidebar HTML (marking the nav item matching `activeId` as `.active`) and `mountSidebars()` fills every `[data-active]` sidebar on load. **When adding a new operator screen, use this mount-point pattern â€” do not hand-write sidebar markup.** The one exception is the Impersonate screen (`s6`), which has a genuinely different client-facing sidebar (different nav items, dashed logo border, "PRIMARK VIEW" tag) and is hardcoded inline instead of mounted.

**Header stat chips.** Page-level KPI numbers live in the `.pg-hdr-bar`, right-aligned next to the title, as compact `.stat-mini` chips inside a `.hdr-stat-row` â€” not as a separate `.stat-row` block under the header. This is the standard pattern for every screen with summary numbers (Retailers, Suppliers, Retailer detail, Impersonate, Products, Packagings, Supplier detail). Keep new "dashboard numbers" consistent with this rather than reintroducing full `.stat` cards in the body.

**Live theme customizer.** A "Theme" button in `.pnav` opens a panel (`#themePanel`) that edits CSS custom properties on `documentElement.style` in real time â€” color pickers for the brand palette, presets, and range sliders for imagery/effects (swoosh opacity, login background opacity/photo toggle, logo sizes, sidebar gradient colors/opacity/angle/spacing, card-glow speed/pause/opacity/colors, spotlight opacity/size/color). Everything it controls is a `:root` variable, so new appearance knobs should also be plain CSS variables consumed by the relevant selector, plus an entry in `EXTRA_CONTROLS` (range/toggle) or `THEME_GROUPS` (color) so they show up in the panel automatically.
- **Export/Import JSON**: `collectThemeState()`/`applyThemeState()` round-trip the whole theme (colors + extra + login-bg toggle + spotlight/sidebar hex colors) to/from a downloadable `.json` file. When adding a new themeable property, wire it into both functions or it won't survive export/import.
- **Login card effects**: the glowing contour (`.login-card::before`) sweeps diagonally (top-left â†’ bottom-right â†’ back) via a JS-generated `@keyframes` (`updateCardGlowAnimation()`) so the speed/pause controls can reshape the animation timing live â€” don't hand-edit the static fallback `@keyframes cardGlow` for behavior changes, edit the JS generator. The cursor-follow spotlight (`.zone-spotlight`) is a real element inside each login zone (not a card-level pseudo), positioned at `z-index:-1` so it glows through the frosted-glass zone background but stays behind the zone's real content (heading/inputs/buttons) â€” replicate this z-index/layering approach for any new glass-hover effects rather than a single card-wide overlay.

**Product/packaging interactivity ported from v5, restyled.** The paginated/sortable/filterable table engine (`ptInit`/`ptRender`/`ptSort`/etc.), the "Add packaging" library modal (`openPkgModal`/`PKG_LIBRARY`), and the country-selector dropdown (`ALL_COUNTRIES`/`selectCountry`) are the same logic as v5, just re-skinned with v6's classes (`.pill-*`, `.chev-btn`, glass `.grp` cards). Reuse these functions for any new data-table or picker UI rather than writing new ad-hoc JS.

## Working in this repo

- To preview a change, just open the `.html` file in a browser â€” opening `file://` works because the shared `css/`+`js/` files are referenced by relative path. For the browser-preview tooling in this environment, serve over HTTP instead (a `.claude/launch.json` defines a `static` config = `python -m http.server 8777`; `preview_start` it, then navigate to `http://localhost:8777/<file>.html`) since relative asset paths + the tooling behave better over HTTP.
- **Cross-cutting look/behaviour changes go in the two shared files** (`css/greenstreets-theme.css`, `js/greenstreets-theme.js`), which all four pages link â€” not in a single page's inline `<style>`. After editing either, bump the `?v=N` query on the `<link>`/`<script>` in all four HTML files (grep `greenstreets-theme`) or a cached copy will hide your change. Page-specific components still live in that page's inline `<style>`.
- The preview tab runs unfocused/hidden, so `element.focus()` doesn't fire real `:focus`/focus events and `requestAnimationFrame` is throttled â€” you can't observe the focus-ring draw live there. Verify it structurally instead: dispatch a bubbling `focusin` to spawn the `.fs-ring`, freeze a frame by setting `--fs-a`, and check the served file contents / easing math.
- Every screen/nav-button/onclick edit is manual HTML â€” grep for the screen's `id="sN"` and its matching `.pnb` button (same index position in the nav bar list) when adding, removing, or reordering screens, since the two lists must stay in sync by position.
- Follow [SKILL.md](SKILL.md) (the `frontend-design` skill) for visual/UX design decisions â€” it governs tone, typography, layout, and copywriting choices for this project and is loaded automatically when doing design work here.
- Given the file sizes (some HTML files exceed 300KB and 1500+ lines), prefer targeted `Grep`/offset-based `Read` over reading a whole file at once, and prefer `Edit` for surgical changes over rewriting a file.
- In v6, watch your `<div>` balance around `<div class="sidebar" data-active="â€¦"></div>` mount points â€” it's an empty element, easy to accidentally leave a stray closing `</div>` next to it when editing (this broke the Retailers screen's layout once: the stray tag closed `.app-body` early, so `.main` ended up as a sibling instead of a flex child and rendered full-width below the sidebar instead of beside it). After structural edits to a v6 screen, sanity-check with a quick browser reload rather than assuming the diff was safe.
- The preview browser tooling in this environment is occasionally flaky (`preview_screenshot` can hang/timeout even when the page is fine) â€” if screenshots stall, fall back to `preview_eval`/`preview_snapshot`/`preview_console_logs` to verify behavior structurally before retrying the screenshot.
